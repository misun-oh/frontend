01

기본 설명

📸
Git은 내 컴퓨터 안에서 코드의 "스냅샷"을 계속 찍어 저장하는 프로그램입니다. GitHub는 그 스냅샷 모음을 인터넷에 올려서 백업하고 협업하게 해주는 웹사이트입니다. VSCode는 이 둘을 터미널 없이 클릭으로 다루게 해주는 에디터입니다.

최초 설정 — 누가 커밋하는지 등록하기

git config --global user.name "본인이름"
git config --global user.email "본인이메일@example.com"
공용 컴퓨터라면 꼭 확인커밋하기 전에 항상 아래로 지금 등록된 이름을 확인하세요. "다른 계정으로 여러 번 커밋해버렸다"는 사고는 대부분 이걸 건너뛰어서 생깁니다. 
git config --global user.name
git config --global user.email

수정 → 스테이징 → 커밋, 세 영역으로 이해하기

Git은 파일의 상태를 세 단계 영역으로 관리합니다. 이 세 영역을 구분해서 이해하면 status/diff가 왜 그렇게 동작하는지 자연스럽게 이해됩니다.

파일 수정          git add            git commit
Working Directory → Staging Area →   Repository
   (작업 디렉토리)      (스테이징 영역)      (저장소, 커밋 완료)
영역의미
Working Directory지금 에디터에서 보고 있는, 실제 파일이 놓인 곳. 여기서 수정하면 "아직 아무 명령도 안 한" 상태
Staging Areagit add로 "이번 커밋에 포함시키겠다"고 찜해둔 대기 공간
Repositorygit commit으로 실제 저장이 완료된 곳 (스냅샷으로 영구 기록됨)

git status — 지금 상태 확인하기

git status

가장 자주 치게 될 명령입니다. 지금 파일들이 세 영역 중 어디에 있는지 한 번에 보여줍니다.

On branch main
Changes to be committed:
        modified:   badge.css          ← Staging Area에 있음 (add 완료)

Changes not staged for commit:
        modified:   card.css           ← Working Directory에만 있음 (add 안 함)

Untracked files:
        newpage.html                   ← Git이 한 번도 추적한 적 없는 새 파일

status 확인 시 알 수 있는 정보들

문구뜻다음 행동
Untracked files한 번도 add한 적 없는 새 파일git add로 추적 시작
Changes not staged for commit이미 추적 중인 파일이 수정됐지만 아직 add 안 함git add로 스테이징
Changes to be committedadd까지 끝나서 커밋 대기 중git commit
nothing to commit, working tree clean아무 변경사항 없음, 전부 커밋 완료 상태할 일 없음

수정 내역 보는 방법 — git diff

명령비교 대상
git diffWorking Directory ↔ Staging Area (스테이징 안 된 수정 내용만)
git diff --stagedStaging Area ↔ Repository (스테이징 된 수정 내용)
diff가 아무것도 안 나올 때가장 흔한 원인은 이미 add까지 끝나서 Changes to be committed 상태이기 때문입니다. 이땐 git diff가 아니라 git diff --staged를 써야 내용이 보입니다. git status로 지금 어느 영역에 있는지 먼저 확인하는 습관을 들이면 헷갈리지 않습니다.

스테이징 되돌리기

add는 했는데 "아직 커밋하긴 이르다" 싶을 때, 파일 내용은 그대로 두고 스테이징만 취소합니다.

git restore --staged 파일명     # 특정 파일만 스테이징 취소
git restore --staged .          # 스테이징된 것 전부 취소
파일 내용은 안 건드림스테이징 취소는 "이번 커밋에 포함시킬지"만 취소하는 거라, 수정한 내용 자체는 그대로 Working Directory에 남습니다. 완전히 예전 버전으로 파일 내용까지 되돌리고 싶다면 3번(되돌리기)에서 배울 reset이 필요합니다.
확인 퀴즈
git status에 Changes to be committed라고 뜬 파일의 수정 내용을 보려면 어떤 명령을 써야 할까요?
정답 보기
git diff --staged입니다. 일반 git diff는 스테이징 안 된 변경사항만 보여주기 때문에, 이미 add된 파일은 --staged 옵션을 붙여야 내용이 보입니다.

HEAD — 지금 내가 어디 있는지 가리키는 포인터

앞으로 HEAD~1 같은 표현이 계속 등장하니, 여기서 미리 짚고 갑니다. (2부에서 배울 "detached HEAD"도 이 개념이 기준입니다)

📍
HEAD는 "지금 내가 커밋 이력의 어느 지점에 있는지"를 가리키는 이정표입니다. 평소엔 브랜치 이름(예: main)을 가리키고 있어서, 새로 커밋할 때마다 그 브랜치가 자동으로 한 칸씩 앞으로 나아갑니다.
HEAD → main → (가장 최근 커밋)
표기의미
HEAD지금 위치 (가장 최근 커밋)
HEAD~1지금 위치에서 1단계 이전 커밋
HEAD~3지금 위치에서 3단계 이전 커밋
3번에서 계속 쓰입니다3번(되돌리기)의 reset --soft HEAD~1이 전부 이 포인터 개념을 기준으로 합니다. 지금은 "HEAD = 현재 위치를 가리키는 화살표"라는 것만 기억해두면 충분합니다.
확인 퀴즈
Git과 GitHub 중, 인터넷 연결 없이도 커밋을 만들 수 있는 건 어느 쪽일까요?
정답 보기
Git입니다. Git은 로컬(내 컴퓨터) 프로그램이라 오프라인에서도 커밋이 가능합니다. GitHub는 온라인 저장소라 push/pull처럼 원격과 통신할 때만 인터넷이 필요합니다.
확인 퀴즈
커밋을 새로 하나 만들면 HEAD는 어떻게 될까요?
정답 보기
HEAD가 가리키는 브랜치(예: main)가 그 새 커밋으로 한 칸 이동합니다. HEAD 자체가 브랜치를 가리키고 있어서, 브랜치가 이동하면 HEAD도 같이 최신 커밋을 가리키게 됩니다.
02

클론 / 초기화 / 원격지 업로드

이미 로컬에 프로젝트가 있을 때

git init                                    # 폴더를 Git 저장소로 만듦
git add .                                   # 모든 파일 스테이징
git commit -m "first commit"
git remote add origin 깃허브저장소주소
git push -u origin main

이미 있는 저장소를 가져올 때

git clone 깃허브저장소주소
init vs cloneinit은 빈 방에서 새로 시작, clone은 이미 있는 저장소를 통째로 복사. 팀 프로젝트 합류 시엔 거의 항상 clone을 씁니다.

연결된 원격 저장소 확인

git remote -v

지금 이 프로젝트가 어느 GitHub 주소와 연결돼있는지 확인합니다. push가 이상한 곳으로 가는 것 같을 때 가장 먼저 확인하는 명령입니다.

.gitignore — 올리지 말아야 할 파일

# .gitignore
node_modules/
.env
.DS_Store
*.log
이미 커밋된 파일은 안 지워짐.gitignore는 "앞으로 추적 안 할 파일"만 지정합니다. 이미 올린 파일은 git rm --cached 파일명으로 따로 빼야 합니다. 그래서 업로드 전에 미리 작성하는 게 중요합니다.
확인 퀴즈
이미 GitHub에 있는 팀 프로젝트에 합류하게 됐습니다. init과 clone 중 어느 걸 써야 할까요?
정답 보기
clone입니다. init은 완전히 새로운 저장소를 만들 때 쓰고, 이미 존재하는 저장소를 내 컴퓨터로 가져올 땐 clone을 씁니다.
03

커밋 내역 확인 & 되돌리기

내역 확인 — log, diff

git log --oneline              # 한 줄씩 간단히
git log --oneline --graph --all # 브랜치까지 그래프로
git diff                        # 아직 커밋 안 한 변경사항 비교
git diff HEAD~1                 # 이전 커밋과 비교

커밋 1개 취소

명령결과
git reset --soft HEAD~1커밋만 취소, 수정 내용은 스테이징 상태로 남음
git reset --hard HEAD~1커밋 + 파일 내용까지 전부 버림 ⚠️
git commit --amend -m "새메시지"취소가 아니라 직전 커밋을 "수정"

여러 개 / 중간 커밋 다루기

git reset --soft HEAD~3     # 최근 3개 한꺼번에 취소
git revert c2b9e10          # 중간 커밋 하나만 콕 집어 취소 (히스토리 보존)
git rebase -i HEAD~5        # 여러 커밋을 하나로 합치기
rebase -i가 하는 일이 명령을 실행하면 최근 5개 커밋 목록이 에디터로 열립니다. 합치고 싶은 커밋 앞의 pick을 squash(줄여서 s)로 바꾸고 저장하면, 그 커밋들이 하나로 뭉쳐집니다. "커밋 5개를 나눠서 했는데 GitHub에 올리기 전에 깔끔하게 정리하고 싶다"는 상황에 씁니다.
이미 push했다면 규칙이 다름협업 중이거나 남이 pull 받았을 수 있다면 revert로 안전하게 취소하세요. rebase는 커밋 히스토리 자체를 다시 쓰는 명령이라 반드시 push 전에만 써야 합니다. 혼자 쓰는 저장소에서 이미 push한 걸 되돌리고 싶다면 reset --hard 후 git push --force를 쓰는데, 이건 원격 저장소의 기록을 강제로 덮어쓰는 명령이라 그 사이 다른 사람이 작업한 내용이 있다면 통째로 날아갈 수 있습니다. 팀 프로젝트에서는 거의 쓰지 않는 게 안전합니다.

원격지로 통째로 맞추기

로컬 작업을 다 버리고 원격 최신 상태로 리셋하고 싶을 때입니다.

git log --oneline              # 뭘 버리게 되는지 먼저 확인
git fetch origin               # 원격 정보만 받아옴 (아직 합치진 않음)
git reset --hard origin/main   # 로컬을 원격 main과 완전히 똑같이 맞춤
흔한 실수git reset --hard HEAD~1은 로컬 커밋 1개만 되돌릴 뿐, 원격과 같아진다는 보장이 없습니다. 원격과 정확히 맞추려면 반드시 origin/main처럼 원격 브랜치를 기준으로 reset해야 합니다.
확인 퀴즈
reset과 revert의 가장 큰 차이는 무엇일까요?
정답 보기
reset은 커밋 자체를 히스토리에서 없애버리는(과거를 지우는) 방식이고, revert는 기존 커밋은 그대로 두고 그걸 되돌리는 새 커밋을 추가하는 방식입니다. 그래서 이미 push해서 다른 사람이 봤을 수도 있는 커밋은 reset이 아니라 revert로 취소하는 게 안전합니다.
04

충돌 해결

원격지와 로컬이 어긋났을 때 선택지는 두 가지입니다 — ①통째로 원격 걸로 맞추기(03에서 배움), 또는 ②직접 충돌을 해결하며 둘 다 살리기.

충돌이 생기는 전형적 상황같은 파일의 같은 줄을 나와 원격 저장소 양쪽에서 다르게 고쳤을 때 발생합니다.

충돌이 나면 파일 안에 이런 표시가 생깁니다.

<<<<<<< HEAD
여기는 "내가(로컬) 고친" 내용
=======
여기는 "원격 저장소에 있던" 내용
>>>>>>> origin/main
1
pull 시도git pull 실행, 충돌 있으면 자동 표시
2
충돌 파일 열기Source Control 탭에 느낌표 아이콘으로 표시됨
3
선택 버튼 클릭Accept Current / Accept Incoming / Accept Both 중 선택
4
기호 사라졌는지 확인<<<<<<< 등이 완전히 지워졌는지 직접 체크
5
다시 커밋해결한 파일을 스테이징 후 커밋하면 끝
확인 퀴즈
<<<<<<< HEAD와 ======= 사이에 있는 내용은 누구 것일까요?
정답 보기
나(로컬)의 내용입니다. =======과 >>>>>>> origin/main 사이의 내용이 원격 저장소의 내용입니다.
05

VSCode Graph 화면 읽는 법 (실전 판독)

Source Control 아이콘 아래쪽의 "그래프(Graph)" 탭을 누르면 지금까지의 커밋 이력이 이렇게 시각적으로 나옵니다. git log --oneline --graph를 클릭 화면으로 옮겨놓은 것과 같습니다.

화면 요소의미
세로로 이어진 파란 점(●)각각 커밋 하나. 점을 잇는 선이 시간 순서(부모→자식 관계)를 나타냄
맨 위, 파란 배경으로 강조된 줄지금 선택된 커밋 (보통 HEAD, 즉 가장 최근 커밋)
main 파란 배지이 커밋이 main 브랜치의 최신 지점(tip)이라는 표시 — 즉 지금 HEAD가 여기 있음
커밋 메시지 옆 이름작성자(author)
선택된 커밋 아래 들여쓰기된 파일 목록그 커밋에서 실제로 바뀐 파일과 변경 종류 (예: index.html ... M)
맨 아래 "Initial commit"이 저장소의 첫 번째 커밋 (부모가 없는 루트 커밋)

파일 옆 알파벳 한 글자의 의미

표시의미
MModified — 기존 파일 수정
AAdded — 새 파일 추가
DDeleted — 파일 삭제
RRenamed — 파일 이름 변경
실전 진단 — 이 커밋 메시지들, 괜찮을까요? 실제 커밋 이력에는 실습, 교안, 정리처럼 한 단어짜리 메시지가 자주 보입니다. 01번에서 배운 "좋은 커밋 메시지" 기준으로 보면, 몇 달 뒤에 이 목록만 보고 "그날 정확히 뭘 고쳤는지" 알 수 있을까요? 알 수 없습니다. "flex 레이아웃 작성"처럼 구체적인 메시지와 비교해보면 차이가 확 드러납니다.
확인 퀴즈
그래프에서 "main" 배지가 붙은 커밋이 이동하려면 어떤 행동을 해야 할까요?
정답 보기
main 브랜치에서 새 커밋을 만들면 됩니다. 커밋할 때마다 HEAD가 가리키는 브랜치가 그 새 커밋으로 이동하므로, main 배지도 그래프에서 새 커밋 쪽으로 따라 올라갑니다.

미션 2 — 커밋 이력 읽기
git log --oneline으로 지금까지의 커밋을 확인하고, 세 번째로 최근인 커밋의 해시를 찾아 적어보세요.
정답 보기
목록에서 위에서 3번째 줄에 있는 커밋의 앞 7자리 해시가 정답입니다. 숫자를 세는 것보다 이렇게 직접 눈으로 찾는 게 실수를 줄입니다.
🌿 응용🏆 25p
미션 3 — 일부러 충돌 만들고 해결하기
짝과 같은 저장소를 각자 clone 받으세요. 같은 파일의 같은 줄을 다르게 고쳐 각자 push를 시도해 충돌을 발생시킨 다음, 04에서 배운 Accept 버튼 방식으로 직접 해결하세요.
정답 보기
먼저 push한 쪽은 성공, 나중 쪽은 거부당합니다. 나중 쪽이 pull을 받으면 충돌이 발생하고, VSCode의 Accept Current/Incoming으로 해결한 뒤 다시 커밋·push하면 됩니다.
🌱 기초🏆 15p
미션 4 — 내 커밋 메시지 진단하기
VSCode Graph 탭(또는 git log --oneline)으로 본인의 실제 커밋 이력을 열어보세요. 가장 부실한 메시지 3개를 찾아서, 그때 실제로 뭘 했는지 기억을 되살려 "무엇을 왜 바꿨는지"가 드러나는 메시지로 다시 써보세요. (실제로 되돌릴 필요는 없고, 종이/메모에 개선안만 적어도 됩니다)
예시 보기
정리 → "카드 컴포넌트 CSS를 basic/components로 분리"
실습 → "flex justify-content 값 4종 비교 실습"
