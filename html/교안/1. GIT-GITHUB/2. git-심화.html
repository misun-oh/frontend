01

브랜치

🌿
브랜치는 원본을 건드리지 않고 평행세계에서 실험하는 것과 같습니다. main은 항상 정상 작동하는 버전으로 유지하고, 새 기능은 별도 브랜치에서 만들다가 잘 되면 합칩니다.
git branch                       # 브랜치 목록
git checkout -b feature/search   # 새 브랜치 만들며 이동
git checkout main                # main으로 돌아가기
git merge feature/search         # main에 합치기
1부 복습 — HEADHEAD는 "지금 내가 커밋 이력의 어느 지점에 있는지" 가리키는 이정표입니다. 평소엔 브랜치 이름(main 등)을 가리킵니다. 자세한 내용은 1부 01번을 참고하세요.
detached HEAD 주의브랜치 이름이 아니라 커밋 해시로 checkout하면 브랜치와의 연결이 끊긴 "detached HEAD" 상태가 됩니다. 이 상태에서 커밋하면 그 커밋이 어떤 브랜치에도 안 속해서 나중에 유실될 수 있습니다. 뭔가 수정하고 커밋할 계획이면 반드시 git checkout -b 새브랜치이름으로 먼저 브랜치를 만들어 붙잡아두세요.

stash — 작업을 잠깐 서랍에 넣어두기

커밋하기엔 애매한 미완성 코드가 있는데 급하게 다른 브랜치로 이동해야 할 때 씁니다.

git stash              # 지금 변경사항을 서랍에 넣고 작업 전 상태로
git checkout main      # 다른 브랜치에서 급한 일 처리
git checkout feature/search
git stash pop           # 서랍에서 다시 꺼내오기 (꺼내면서 서랍은 비워짐)
확인 퀴즈
커밋하기엔 애매한 미완성 코드를 그대로 두고 다른 브랜치로 급히 이동해야 합니다. 뭘 써야 할까요?
정답 보기
git stash 입니다. 미완성 코드를 임시로 보관해두고 깨끗한 상태로 브랜치를 이동한 뒤, 나중에 stash pop으로 다시 꺼내올 수 있습니다.
확인 퀴즈
브랜치 이름이 아니라 커밋 해시로 checkout했을 때 생기는 위험한 상태의 이름은?
정답 보기
detached HEAD입니다. 이 상태에서 커밋하면 어떤 브랜치에도 속하지 않아 나중에 그 커밋을 잃어버릴 수 있습니다.
02

GitHub 협업

지금까지는 Git 자체 기능이었다면, 여기부터는 GitHub만의 협업 기능입니다.

기능역할
Fork다른 사람의 저장소를 내 계정으로 통째로 복사해서 내 소유로 만듦 (오픈소스에 기여할 때 주로 사용)
Pull Request(PR)"이 변경사항을 원본 저장소에 반영해주세요"라고 요청하는 것. 리뷰를 거쳐 merge됨
Issue버그 리포트, 기능 요청 등을 기록하는 게시판 같은 기능

팀 프로젝트에서의 전형적인 흐름

1
브랜치 생성git checkout -b feature/login
2
작업 후 push내 브랜치를 원격에 올림
3
Pull Request 열기GitHub 웹사이트에서 "이 브랜치를 main에 합쳐주세요" 요청 생성
4
코드 리뷰팀원이 코드를 보고 댓글/승인
5
Merge승인되면 main 브랜치에 합쳐짐
오픈소스 기여는 한 단계 더 있음남의 저장소는 직접 push할 권한이 없으므로, Fork(내 계정으로 복사) → Clone(내 컴퓨터로) → 브랜치 작업 → 내 Fork에 push → 원본 저장소로 Pull Request 순서를 거칩니다.
확인 퀴즈
권한이 없는 오픈소스 프로젝트에 기여하고 싶습니다. 가장 먼저 해야 할 것은?
정답 보기
Fork입니다. 직접 push할 권한이 없으니 먼저 그 저장소를 내 계정으로 복사(Fork)한 뒤, 그걸 clone해서 작업하고 Pull Request를 보내야 합니다.
03

명령어 총정리 (치트시트)

2부에서 배운 명령과, GitHub 웹사이트에서 클릭으로 하는 협업 작업을 정리했습니다.

분류명령 / 작업설명
브랜치git branch브랜치 목록
git checkout -b [이름]브랜치 생성+이동
git merge [브랜치]현재 브랜치에 합치기
git stash / git stash pop작업 임시 보관/꺼내기
GitHub 웹에서Fork남의 저장소를 내 계정으로 복사
Pull Request브랜치 변경사항을 main에 합쳐달라고 요청
Issue버그/기능요청 기록
1부 치트시트도 확인하세요최초 설정, 시작하기, 기본 흐름, 확인하기, 되돌리기 관련 명령은 1부 치트시트에 정리돼 있습니다.
★

종합 미션

2부에서 배운 걸 종합해서 실제 상황처럼 풀어봅니다.

🌿 응용🏆 20p
미션 1 — 브랜치 + stash 조합 실습
feature/dark-mode 브랜치를 만들어 배경색을 바꾸다가, 다 마치지 않은 채로 git stash로 임시 보관하고 main으로 이동해보세요. 다시 브랜치로 돌아와 git stash pop으로 작업을 이어가세요.
정답 보기 
git checkout -b feature/dark-mode
# 배경색 일부만 수정
git stash
git checkout main
git checkout feature/dark-mode
git stash pop
# 아까 하던 수정이 그대로 돌아와 있어야 정상
🌳 도전🏆 25p
미션 2 — detached HEAD 직접 겪어보기
git log --oneline으로 과거 커밋 해시를 확인 후 git checkout [해시]로 이동하고 경고 메시지를 확인하세요. 이 상태에서 파일을 수정·커밋한 다음 git checkout main으로 돌아가면 그 커밋이 어떻게 되는지 확인하고, 잃지 않으려면 어떻게 했어야 하는지 설명하세요.
정답 보기
main으로 돌아가면 detached 상태에서 만든 커밋은 어느 브랜치에도 안 보입니다 (완전히 사라진 건 아니고, Git이 내부적으로 남겨두는 reflog라는 별도 이력으로 당분간 복구는 가능하지만, 방치하면 결국 정리됩니다). 커밋 전에 git checkout -b 임시브랜치로 먼저 브랜치를 만들어뒀어야 안전하게 보존됐을 것입니다.
🌳 도전🏆 20p
미션 3 — 미니 PR 흐름 체험
짝과 함께 같은 저장소에서, 한 명은 feature/badge-color 브랜치로 뱃지 색을 바꾸고 push한 뒤 GitHub에서 실제로 Pull Request를 열어보세요. 다른 한 명은 리뷰어가 되어 코드에 댓글을 달고 승인(Approve)한 뒤 Merge까지 진행하세요.
정답 보기
GitHub 저장소 페이지에서 "Compare & pull request" 버튼 → 제목/설명 작성 → Create pull request → 리뷰어가 Files changed 탭에서 댓글 → Review changes → Approve → Merge pull request 순서로 진행하면 됩니다.

🌱 브랜치 릴레이 + 미니 데모데이

① 4명이 한 팀. main에서 각자 다른 브랜치(feature/a, feature/b...)를 만들어 서로 다른 부분(뱃지 색, 카드 그림자, 버튼 모양 등)을 동시에 작업합니다. ② 각자 Pull Request를 열고, 팀원끼리 돌아가며 리뷰·승인·머지까지 진행합니다. ③ 마지막엔 4개의 변경사항이 모두 합쳐진 main을 다 함께 확인하는 미니 데모데이로 마무리합니다.
