10-1

JSON — 서버와 주고받는 데이터 형식

JSON(JavaScript Object Notation)은 자바스크립트 객체 표기법을 본뜬 텍스트 데이터 형식입니다. 서버와 데이터를 주고받을 때의 사실상 표준이며, 키를 반드시 큰따옴표로 감쌉니다.

메서드방향설명
JSON.stringify(객체)객체 → JSON 문자열저장·전송용으로 변환 (7장 localStorage에서 사용)
JSON.parse(문자열)JSON 문자열 → 객체받은 데이터를 다시 객체로 복원
const movie = { title: "도시의 밤", rating: 9.8 };

const jsonText = JSON.stringify(movie);
console.log(jsonText);          // '{"title":"도시의 밤","rating":9.8}' (문자열)

const restored = JSON.parse(jsonText);
console.log(restored.title);    // 도시의 밤 (다시 객체)
10-2

에러 처리 — try / catch / finally / throw

실행 중 오류가 나면 프로그램은 그 자리에서 멈춥니다. try 블록에서 시도하고, 오류가 나면 catch 블록이 받아서 처리하면 멈추지 않고 이어갈 수 있습니다. finally는 성공하든 실패하든 항상 실행됩니다.

try {
  const data = JSON.parse("이건 JSON이 아님");  // 여기서 오류 발생!
  console.log("이 줄은 실행되지 않음");
} catch (error) {
  console.log("오류를 잡았습니다:", error.message);
} finally {
  console.log("성공하든 실패하든 실행");
}

// throw: 직접 오류를 발생시키기
function setRating(n) {
  if (n < 0 || n > 10) throw new Error("평점은 0~10 사이여야 합니다");
  return n;
}
⭐ 네트워크 코드에는 에러 처리가 필수

외부 서버는 언제든 느려지거나 실패할 수 있습니다. fetch를 쓰는 코드는 항상 try/catch로 감싸고, 실패 시 사용자에게 안내 문구를 보여주는 것까지가 한 세트입니다.

10-3

동기 vs 비동기, 그리고 Promise

동기(sync)는 한 줄이 끝나야 다음 줄로 넘어가는 방식, 비동기(async)는 오래 걸리는 작업을 걸어두고 기다리지 않고 다음 코드를 실행하는 방식입니다.

비동기로 동작하는 작업은 "시간이 걸리는 일"이라는 공통점이 있습니다. setTimeout은 그중 하나의 예시일 뿐이고, 실제로는 아래처럼 다양한 상황에서 비동기가 등장합니다.

종류구체적인 예왜 비동기인가
타이머setTimeout, setInterval (07장)정해진 시간이 지나야 실행되므로 그때까지 기다리지 않음
사용자 이벤트addEventListener (09장)사용자가 언제 클릭할지 알 수 없어, 클릭이 일어날 때까지 기다리지 않고 등록만 해둠
네트워크 요청fetch (이번 장)서버 응답이 얼마나 걸릴지 알 수 없음 — 0.1초일 수도, 5초일 수도
파일 · 미디어 로딩이미지, 오디오/비디오 로딩파일 용량과 네트워크 속도에 따라 로딩 시간이 다름
브라우저 저장소localStorage 등 (07장)이 과정에서는 사실 동기지만, 실무의 더 큰 저장소(IndexedDB 등)는 비동기로 동작
⭐ 공통점: '언제 끝날지 미리 알 수 없는 일'은 비동기로 처리합니다

반대로 1 + 2 같은 계산은 시간이 예측 가능(사실상 즉시)하므로 동기로 처리해도 문제없습니다. "얼마나 걸릴지 모르는 일 때문에 나머지 코드가 멈춰서는 안 된다"는 것이 비동기가 필요한 이유입니다.

아래는 07장에서 이미 다룬 setTimeout으로 비동기의 실행 순서를 다시 확인하는 예제입니다.

console.log("1. 시작");

setTimeout(function () {
  console.log("2. 3초 후 실행 (비동기)");
}, 3000);

console.log("3. 끝");

// 실행 순서: 1 → 3 → 2   (숫자 순서가 아님!)
// setTimeout은 "3초 뒤에 실행해줘"라고 예약만 해두고,
// 기다리지 않고 바로 다음 줄(3)로 넘어가기 때문
실행 결과
1. 시작
3. 끝
2. 3초 후 실행 (비동기)  ← 3초 뒤에 뒤늦게 출력

Promise는 「나중에 결과를 주겠다는 약속」 객체로, 세 가지 상태를 가집니다.

상태의미이어지는 처리
pending대기 중 (결과 아직 없음)—
fulfilled성공 (결과 도착).then(콜백) 실행
rejected실패 (오류 발생).catch(콜백) 실행
10-4

콜백 방식의 한계 — 콜백 지옥

04장에서 배운 콜백은 비동기 작업 하나를 처리할 땐 편리하지만, 순서 있는 여러 비동기 작업(방금 배운 타이머·이벤트·네트워크 요청 등)을 이어 붙이면 코드가 급격히 지저분해집니다. 아래는 타이머(setTimeout)로 예를 들지만, fetch 같은 다른 비동기 작업을 연달아 쓸 때도 똑같은 문제가 생깁니다.

// "1단계 끝나면 2단계, 2단계 끝나면 3단계" 를 콜백으로 표현하면...
function step1(callback) {
  setTimeout(function () {
    console.log("1단계: 회원 정보 조회");
    callback();
  }, 1000);
}

function step2(callback) {
  setTimeout(function () {
    console.log("2단계: 결제 처리");
    callback();
  }, 1000);
}

function step3() {
  setTimeout(function () {
    console.log("3단계: 완료 알림");
  }, 1000);
}

// 순서대로 실행하려면 콜백 안에 콜백을 계속 중첩해야 함
step1(function () {
  step2(function () {
    step3();
  });
});
// 4단계, 5단계가 더 늘어나면? 오른쪽으로 계속 밀려나는 "콜백 지옥"
⭐ 콜백 지옥(Callback Hell)

비동기 작업이 늘어날수록 코드가 오른쪽으로 계속 들여쓰기 되며 피라미드 모양이 되는 현상을 콜백 지옥이라 부릅니다. 에러 처리도 각 단계마다 따로 해야 해서 실수하기 쉽습니다. 바로 위에서 배운 Promise가 이 문제를 해결하고, Promise를 더 읽기 쉽게 쓰는 문법이 async/await입니다.

10-5

Promise를 직접 만들기

지금까지는 fetch가 이미 만들어서 리턴해주는 Promise를 받기만 했습니다. Promise가 정확히 어떻게 동작하는지 이해하려면 직접 만들어보는 것이 가장 빠릅니다. new Promise(function(resolve, reject) {...}) 형태로 만들고, 성공하면 resolve(값), 실패하면 reject(에러)를 호출합니다.

function delay(ms, value) {
  return new Promise(function (resolve, reject) {
    if (ms < 0) {
      reject(new Error("시간은 음수일 수 없습니다"));
      return;
    }
    setTimeout(function () {
      resolve(value);   // 이 값이 .then()의 매개변수로 전달됨
    }, ms);
  });
}

delay(1000, "완료!").then(function (result) {
  console.log(result);   // 1초 후 "완료!"
});

delay(-1, "x").catch(function (error) {
  console.log("에러:", error.message);   // 즉시 "에러: 시간은 음수일 수 없습니다"
});

이 원리를 알면, fetch(...)도 결국 내부에서 서버 응답이 오면 resolve, 네트워크가 끊기면 reject를 호출하는 Promise일 뿐이라는 것이 이해됩니다.

⭐ resolve/reject는 딱 한 번만 유효합니다

Promise 하나는 성공 또는 실패 중 하나로 딱 한 번만 결정됩니다. resolve를 부른 뒤 reject를 또 불러도 무시됩니다 — 콜백은 실수로 여러 번 호출되는 버그가 생길 수 있는 반면, Promise는 구조적으로 그런 위험이 없다는 것도 장점입니다.

10-6

fetch — then 체이닝 방식

fetch(주소)는 HTTP 요청을 보내고 Promise를 리턴합니다. 응답(response)이 오면 response.json()으로 본문을 객체로 변환하는데, 이것도 Promise라 then을 한 번 더 씁니다.

fetch("https://jsonplaceholder.typicode.com/users")
  .then(function (response) {
    if (!response.ok) throw new Error("HTTP 오류 " + response.status);
    return response.json();      // 본문 → 객체 (이것도 Promise)
  })
  .then(function (data) {
    console.log(data);            // 드디어 진짜 데이터!
  })
  .catch(function (error) {
    console.log("요청 실패:", error.message);
  });
🙋 학생들이 자주 틀리는 부분 — 404 응답인데도 catch로 안 가서 놓치는 실수
❌ 이렇게 쓰면 안 됩니다 
fetch("https://jsonplaceholder.typicode.com/users/9999")   // 존재하지 않는 id
  .then(function (response) {
    return response.json();   // response.ok 확인 없이 바로 파싱 시도
  })
  .then(function (data) {
    console.log(data.name);   // 데이터가 없는데 이름을 꺼내려다 undefined 관련 오류
  })
  .catch(function (error) {
    console.log("실패:", error.message);  // 여기로 안 옴 — 404도 "응답은 받았음"
  });
✅ 이렇게 고칩니다 
fetch("https://jsonplaceholder.typicode.com/users/9999")
  .then(function (response) {
    if (!response.ok) throw new Error("HTTP 오류 " + response.status);  // 직접 확인!
    return response.json();
  })
  .then(function (data) {
    console.log(data.name);
  })
  .catch(function (error) {
    console.log("실패:", error.message);   // "실패: HTTP 오류 404" ✅
  });

fetch의 Promise는 네트워크 자체가 끊기는 등 요청이 아예 실패했을 때만 reject됩니다. 404·500처럼 서버가 응답은 했지만 오류를 알리는 경우는 '성공적으로 응답을 받은 것'으로 취급되어 then으로 들어옵니다. 그래서 response.ok를 직접 확인해서 throw해줘야만 catch가 제대로 작동합니다.

연습용 무료 서버: https://jsonplaceholder.typicode.com — /users(회원 10명), /posts(글 100개) 등을 응답해주는 테스트 전용 API입니다.

10-7

async / await — 비동기를 동기처럼 읽기

async를 붙인 함수 안에서는 await로 Promise의 결과를 기다렸다가 변수에 담을 수 있어, then 체인보다 훨씬 읽기 쉽습니다. 에러는 try/catch로 잡습니다.

async function loadUsers() {
  try {
    const response = await fetch("https://jsonplaceholder.typicode.com/users");
    if (!response.ok) throw new Error("HTTP 오류 " + response.status);

    const data = await response.json();
    console.log(data);
  } catch (error) {
    console.log("요청 실패:", error.message);
  }
}

loadUsers();
⭐ await는 async 함수 안에서만

함수 밖(전역)에서 await를 쓰면 일반 script에서는 오류입니다. 「async 함수를 만들고 → 그 안에서 await」 순서를 몸에 붙이세요. 이 과정의 표준 패턴은 async 함수 + await fetch + try/catch + response.ok 확인 4종 세트입니다.

🔗 index.html 연결 포인트

받은 데이터가 객체 배열이기만 하면, 8장의 renderMovies 파이프라인에 그대로 흘려보낼 수 있습니다. 「fetch → 배열 → render」 — 데이터의 출처만 바뀔 뿐 그리는 방법은 동일합니다.

10-8

여러 비동기 작업을 동시에 처리하기 — Promise.all

await를 여러 번 순서대로 쓰면, 앞의 요청이 끝나야 다음 요청이 시작됩니다. 서로 관련 없는 요청들이라면 굳이 순서대로 기다릴 필요가 없습니다 — Promise.all([...])로 한꺼번에 시작해서 전부 끝나면 한 번에 결과를 받을 수 있습니다.

// ❌ 순차 처리 — 각 요청이 1초씩 걸린다면 총 3초
async function loadSequential() {
  const users = await fetch("https://jsonplaceholder.typicode.com/users").then(r => r.json());
  const posts = await fetch("https://jsonplaceholder.typicode.com/posts").then(r => r.json());
  const todos = await fetch("https://jsonplaceholder.typicode.com/todos").then(r => r.json());
  console.log(users.length, posts.length, todos.length);
}

// ✅ 동시 처리 — 3개를 한꺼번에 요청, 총 시간은 가장 느린 것 하나만큼(약 1초)
async function loadParallel() {
  const [users, posts, todos] = await Promise.all([
    fetch("https://jsonplaceholder.typicode.com/users").then(r => r.json()),
    fetch("https://jsonplaceholder.typicode.com/posts").then(r => r.json()),
    fetch("https://jsonplaceholder.typicode.com/todos").then(r => r.json()),
  ]);
  console.log(users.length, posts.length, todos.length);
}

Promise.all의 결과 배열은 05장에서 배운 배열 구조분해할당으로 바로 이름을 붙여 받을 수 있습니다 — 위 코드의 const [users, posts, todos] = ... 부분입니다.

⚠️ Promise.all은 하나라도 실패하면 전체가 실패합니다

세 요청 중 하나라도 reject되면, 나머지가 성공했어도 Promise.all 전체가 그 순간 바로 reject되어 catch로 빠집니다. "일부만 성공해도 괜찮은" 상황에는 더 심화된 Promise.allSettled를 쓰지만, 이 과정에서는 "모두 성공해야만 의미 있는" 상황에만 Promise.all을 사용합니다.

10-9

HTTP 메서드와 fetch 옵션 정리

지금까지 fetch로 데이터를 조회(GET)만 했습니다. 서버와 주고받는 작업에는 목적에 따라 다른 HTTP 메서드가 있습니다.

메서드의미이 과정에서
GET데이터 조회 — 지금까지 쓴 fetch가 전부 GET (기본값)공공데이터·포켓몬 도감 조회에 사용
POST새 데이터 생성 — 서버에 정보를 새로 등록할 때STT 오디오 전송, TTS 텍스트 전송에 사용
PUT기존 데이터 전체 수정이 과정에서는 다루지 않음(참고만)
DELETE데이터 삭제이 과정에서는 다루지 않음(참고만)

GET 외의 메서드를 쓰려면 fetch의 두 번째 인자(옵션 객체)에 method, headers, body를 지정합니다. 10-11의 TTS 코드가 이미 이 형태를 쓰고 있었습니다.

fetch("https://api.example.com/movies", {
  method: "POST",                              // GET이 아닌 POST 사용
  headers: {
    "Content-Type": "application/json",        // "본문이 JSON 형식"임을 서버에 알림
  },
  body: JSON.stringify({ title: "새 영화", rating: 9.0 }),  // body는 항상 문자열!
});
🙋 학생들이 자주 틀리는 부분 — Content-Type 헤더를 안 맞춰서 서버가 데이터를 못 읽음
❌ 이렇게 쓰면 안 됩니다 
// body는 객체를 JSON 문자열로 바꿔 보냈는데, 헤더 설정을 깜빡함
fetch("https://api.example.com/movies", {
  method: "POST",
  body: JSON.stringify({ title: "새 영화" }),
  // headers 없음!
});
// 서버가 body를 "일반 텍스트"로 오해해서 JSON으로 못 읽고 오류 응답을 줄 수 있음
✅ 이렇게 고칩니다 
fetch("https://api.example.com/movies", {
  method: "POST",
  headers: { "Content-Type": "application/json" },   // "이 본문은 JSON입니다" 명시
  body: JSON.stringify({ title: "새 영화" }),
});

body에 무엇을 담아 보내든, 서버는 그것을 어떻게 해석할지 headers의 Content-Type를 보고 판단합니다. JSON 문자열을 보내면서 Content-Type을 빠뜨리면, 서버가 파싱에 실패해 오류를 돌려주는 경우가 많습니다. 예외적으로 10-11의 FormData는 Content-Type을 직접 쓰지 않는다는 것도 함께 기억하세요(브라우저가 자동 설정).

10-10

공공데이터포털 API 활용

공공데이터포털(data.go.kr)은 정부·공공기관 데이터를 API로 제공합니다. 이용 절차: ① 회원가입 → ② 원하는 데이터 검색 → ③ 활용신청 → ④ 발급된 인증키(serviceKey)로 호출.

요청 URL은 「기본주소 + ?serviceKey=인증키 + &매개변수들」 형태의 쿼리 스트링으로 구성합니다. (7장 location.search에서 본 ?키=값&키=값 형식)

// 예: 박스오피스 순위 조회 (영화진흥위원회 KOBIS 오픈API)
const KEY = "발급받은_인증키";
const date = "20260712";  // 조회할 날짜 (어제)
const url = `https://www.kobis.or.kr/kobisopenapi/webservice/rest/boxoffice/searchDailyBoxOfficeList.json?key=${KEY}&targetDt=${date}`;

async function loadBoxOffice() {
  try {
    const response = await fetch(url);
    if (!response.ok) throw new Error("HTTP 오류 " + response.status);
    const data = await response.json();

    const list = data.boxOfficeResult.dailyBoxOfficeList; // 응답 구조는 문서 확인!
    list.forEach(function (m) {
      console.log(`${m.rank}위 ${m.movieNm} (관객 ${m.audiCnt}명)`);
    });
  } catch (error) {
    console.log("불러오기 실패:", error.message);
  }
}
⚠️ 응답 구조는 반드시 문서/콘솔로 확인

API마다 데이터를 감싸는 구조가 다릅니다. 먼저 console.log(data)로 전체를 찍어 어디에 배열이 들어있는지 확인한 뒤 꺼내는 습관을 들이세요. 또한 브라우저에서 직접 호출 시 CORS 오류가 나는 API도 있습니다 — 서버가 브라우저의 교차 출처 요청을 허용해야 하며, 막혀 있으면 중간 서버(프록시)가 필요합니다.

10-11

GPT 음성 API — STT와 TTS

OpenAI의 오디오 API를 쓰면 STT(Speech-to-Text, 음성→텍스트)와 TTS(Text-to-Speech, 텍스트→음성)를 몇 줄로 구현할 수 있습니다. 요청 시 발급받은 API 키를 Authorization 헤더에 담습니다. (모델명은 변할 수 있으니 공식 문서에서 최신 확인)

TTS — 텍스트를 음성으로: 응답이 JSON이 아니라 오디오 파일(blob)이라는 점이 핵심입니다.

async function speak(text) {
  const response = await fetch("https://api.openai.com/v1/audio/speech", {
    method: "POST",
    headers: {
      "Authorization": "Bearer " + OPENAI_KEY,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      model: "tts-1",
      voice: "nova",
      input: text,
    }),
  });

  const audioBlob = await response.blob();          // JSON이 아니라 blob!
  const audioUrl = URL.createObjectURL(audioBlob);  // blob → 재생 가능한 주소
  new Audio(audioUrl).play();
}

speak("오늘의 추천 영화는 도시의 밤입니다");

STT — 음성을 텍스트로: 지금까지 fetch의 body에는 JSON.stringify(...)로 만든 문자열을 넣었습니다. 하지만 오디오 파일처럼 텍스트가 아닌 파일 데이터를 보낼 때는 FormData라는 전용 객체를 씁니다.

FormData 메서드설명
new FormData()빈 폼 데이터 객체 생성
formData.append(이름, 값)이름-값 쌍을 추가 (파일도 값으로 넣을 수 있음)
const formData = new FormData();
formData.append("file", audioFile);   // 파일 데이터
formData.append("model", "whisper-1"); // 일반 값도 함께 담을 수 있음

console.log(formData);   // 콘솔에는 내용이 잘 안 보이지만, 서버로는 정상 전송됨

마이크 녹음(MediaRecorder)으로 얻은 오디오 파일을 FormData에 담아 fetch로 전송합니다.

async function transcribe(audioFile) {
  const formData = new FormData();
  formData.append("file", audioFile);      // 녹음된 오디오 파일
  formData.append("model", "whisper-1");

  const response = await fetch("https://api.openai.com/v1/audio/transcriptions", {
    method: "POST",
    headers: { "Authorization": "Bearer " + OPENAI_KEY },
    body: formData,   // FormData는 Content-Type을 직접 쓰지 않음(자동 설정)
  });

  const data = await response.json();
  console.log("인식된 텍스트:", data.text);
  return data.text;
}
⚠️ API 키를 프론트 코드에 넣으면 안 됩니다 (수업용 예외)

브라우저의 자바스크립트는 누구나 소스 보기로 열람할 수 있어, 키를 넣는 순간 도용됩니다(과금 폭탄!). 실제 서비스에서는 키를 서버에 두고 브라우저는 서버를 경유합니다. 이 수업에서는 원리 이해를 위해 예외적으로 키를 직접 넣되, 실습 후 즉시 키를 폐기(재발급)하세요. 유료 API이므로 강사가 안내하는 실습용 키만 사용합니다.
