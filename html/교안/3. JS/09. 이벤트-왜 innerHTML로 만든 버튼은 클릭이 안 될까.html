"왜 innerHTML로 만든 버튼은 클릭이 안 될까?"

이벤트 흐름(Bubbling/Capturing)의 이해와 효율적인 동적 DOM 제어(Event Delegation)

1. 학습 목표

본 교안은 웹 개발 과정에서 가장 빈번하게 발생하는 '동적 생성 요소의 이벤트 미작동 문제'를 진단하고, 이를 브라우저의 이벤트 흐름 메커니즘을 통해 완벽히 이해하는 것을 목표로 합니다.

  • innerHTML 등을 통해 사후에 생성된 요소에 이벤트가 발생하지 않는 근본적인 원인을 설명할 수 있다.
  • 브라우저의 이벤트 전달 3단계 흐름(Capturing → Target → Bubbling)을 이해한다.
  • 이벤트 위임(Event Delegation) 패턴을 활용하여 동적 요소를 효율적으로 제어하고 메모리를 최적화할 수 있다.

2. 문제 제기: 사라진 이벤트 리스너

JavaScript를 활용해 화면을 갱신할 때, 우리는 흔히 innerHTML을 사용하여 HTML 문자열을 주입합니다. 하지만 이때 이전에 연결해 둔 이벤트가 동작하지 않는 기현상이 발생합니다.

🚨 흔한 실수 시나리오:
1. 페이지가 처음 로드되었을 때 document.querySelectorAll('.delete-btn')로 모든 삭제 버튼을 찾아 이벤트를 등록합니다.
2. 사용자가 [추가] 버튼을 누르면 container.innerHTML += '...<button class="delete-btn">삭제</button>' 코드가 실행됩니다.
3. 결과: 새로 추가된 삭제 버튼을 아무리 클릭해도 아무런 반응이 없습니다.

왜 이런 현상이 발생할까요?

  1. 평가 시점의 한계: addEventListener 코드가 실행되는 시점에는 화면에 '기존 요소'만 존재합니다. 브라우저는 당시 존재하던 노드들에게만 이벤트를 인쇄하듯 붙여놓고 지나갑니다.
  2. innerHTML의 파괴적 특성: innerHTML을 변경하면 기존의 자식 노드들이 브라우저 메모리에서 완전히 삭제된 후 새로운 HTML 문자열을 기반으로 **처음부터 다시 생성**됩니다. 이 과정에서 기존 노드에 걸려있던 이벤트 리스너들까지 함께 유실됩니다.

3. 브라우저의 이벤트 전달 메커니즘 (3가지 단계)

이 문제를 우아하게 해결하기 위해서는 브라우저가 이벤트를 어떻게 전달하는지 그 시스템을 알아야 합니다. 어떤 요소를 클릭했을 때, 브라우저는 해당 요소만 콕 집어 이벤트를 실행하지 않고 최상위 부모로부터 U자 형태로 순회합니다.

단계 (Phase) 방향 (Direction) 설명
1. 캡처링 (Capturing) 최상위 부모 → 자식 (하향식) window에서 시작하여 이벤트가 실제 발생한 타깃 요소를 향해 내려가는 단계
2. 타깃 (Target) 실제 클릭된 요소 도착 이벤트가 목적지에 도달하여 해당 요소에 직접 등록된 리스너를 호출하는 단계
3. 버블링 (Bubbling) 자식 → 최상위 부모 (상향식) 이벤트가 다시 부모들을 거쳐 물속의 거품처럼 최상위 window까지 솟구치는 단계
💡 핵심 포인트: addEventListener의 기본 동작

JavaScript의 element.addEventListener('click', handler)는 기본적으로 3단계인 버블링(Bubbling) 단계에서 올라오는 이벤트를 감지합니다. 만약 1단계인 캡처링 단계에서 이벤트를 낚아채고 싶다면 세 번째 인자에 { capture: true } 옵션을 제공해야 합니다.

4. 해결책: 이벤트 위임 (Event Delegation)

자식 요소가 언제 생성될지 모르거나 수시로 바뀐다면, "언제나 그 자리에 고정되어 있는 변하지 않는 부모 요소"에게 이벤트를 딱 한 번만 위임하는 테크닉을 사용합니다. 자식에게 발생한 이벤트가 버블링을 통해 부모에게 전달되는 특성을 이용하는 것입니다.

이벤트 위임 구현의 핵심 로직

// 1. 동적으로 요소를 담을 고정된 부모 요소를 선택합니다.
const parentList = document.querySelector('#todo-list');

// 2. 부모에게 단 하나의 이벤트 리스너만 설정합니다.
parentList.addEventListener('click', function(event) {
    // event.target은 실제 클릭된 가장 안쪽의 요소를 가리킵니다.
    
    // 3. 원하는 자식 요소(예: 삭제 버튼)가 맞는지 조건문으로 필터링합니다.
    if (event.target.classList.contains('delete-btn')) {
        // 4. 매칭된다면 실제 수행할 로직을 실행합니다.
        const targetRow = event.target.closest('li');
        targetRow.remove();
        console.log('동적 요소 삭제 완료!');
    }
});

5. 이벤트 위임 패턴의 핵심 장점

  • 동적 요소 완벽 제어: innerHTML이나 Ajax 통신, 오픈 API 등을 통해 자식 요소가 아무리 불규칙하게 추가·삭제되어도 JavaScript 코드를 재등록할 필요가 없습니다.
  • 메모리 절약 및 성능 향상: 자식 요소가 1,000개라면 원래 1,000개의 이벤트 리스너를 메모리에 올려야 하지만, 이벤트 위임을 쓰면 부모 요소 단 1개의 리스너만 메모리에 유지하므로 성능상 매우 유리합니다.
  • 코드 유지보수성 증대: 이벤트 생성 및 제거 로직이 한 곳(부모 리스너)에 모여있어 디버깅과 코드 관리가 직관적입니다.
📝 요약 및 결론

"왜 innerHTML로 만든 버튼은 클릭이 안 될까?"에 대한 답은 **"버튼이 태어날 때 이벤트 리스너를 받지 못했고, 기존 리스너는 파괴되었기 때문"**입니다. 이를 해결하기 위해 우리는 이벤트가 부모로 거슬러 올라가는 **이벤트 버블링** 메커니즘을 활용하여, 상위 부모에게 이벤트를 대행시키는 **이벤트 위임**을 구현해야 합니다.
