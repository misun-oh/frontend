동기(Synchronous) vs 비동기(Asynchronous)

자바스크립트는 원래 코드를 위에서 아래로, 한 줄씩 순서대로 실행합니다. 이걸 동기 방식이라고 합니다.

sync.js 
console.log("1번");
console.log("2번");
console.log("3번");

// 결과: 1번 → 2번 → 3번 (순서 그대로)

그런데 2번 작업이 서버 요청처럼 오래 걸리는 작업이라면 어떨까요? 그 작업을 기다리지 않고 다음 코드를 먼저 실행한 뒤, 작업이 끝나면 결과를 나중에 처리하는 방식이 비동기입니다.

async.js 
console.log("1번");

setTimeout(() => {
  console.log("2번 (3초 후 실행)");
}, 3000);

console.log("3번");

// 결과: 1번 → 3번 → 2번 (3초 후 실행)
💡
비유하자면 — 식당에서 주문을 하면(비동기 요청) 요리가 나올 때까지 서서 기다리지 않고 자리에 앉아 다른 일을 합니다(다음 코드 실행). 요리가 완성되면 직원이 가져다줍니다(콜백 실행). 이런 비동기 처리가 특히 필요한 대표적 상황이 바로 서버와 데이터를 주고받는 통신입니다.
01

콜백(Callback)과 콜백 지옥

비동기 통신의 가장 원초적인 해결책, 그리고 그 대가

콜백 함수란?

다른 함수에게 인자로 넘겨주는 함수입니다. "이 작업이 끝나면 이 함수를 실행해줘"라고 부탁하는 함수라고 생각하면 쉽습니다.

callback-basic.js
function 인사하기(이름, 콜백함수) {
  console.log(`${이름}님 안녕하세요`);
  콜백함수(); // 전달받은 함수를 실행
}

인사하기("철수", function () {
  console.log("반갑습니다!");
});

// 철수님 안녕하세요
// 반갑습니다!

콜백 지옥이란?

비동기 작업을 순서대로 여러 번 처리해야 할 때 문제가 생깁니다. 사용자 정보 → 주문 목록 → 주문 상세 → 배송 정보를 차례로 가져와야 한다면, 콜백으로는 이렇게 됩니다.

피라미드 오브 둠, 직접 펴보세요 → 중첩된 콜백
1가져오기_사용자정보("user1", function (사용자) {
2가져오기_주문목록(사용자.id, function (주문목록) {
3가져오기_주문상세(주문목록[0].id, function (주문상세) {
4가져오기_배송정보(주문상세.id, function (배송정보) {
5console.log("배송:", 배송정보.status);
6});
7});
8});
9});
▲ 오른쪽으로 계속 밀려나는 상태: REJECTED (읽기 힘듦)
⚠️
이렇게 콜백 안에 콜백이 계속 중첩되어 오른쪽으로 밀려나는 모양을 콜백 지옥(Callback Hell), 혹은 피라미드처럼 생겼다 해서 피라미드 오브 둠(Pyramid of Doom)이라 부릅니다.

콜백 지옥의 문제점

문제점설명
가독성 저하코드가 옆으로 계속 밀려나서 읽기 힘들어짐
에러 처리 어려움각 단계마다 에러 처리를 따로 해줘야 함
유지보수 어려움순서를 바꾸거나 로직을 수정하기 매우 번거로움
디버깅 어려움어느 단계에서 문제가 생겼는지 추적하기 힘듦
02

Promise

"미래에 완료될 작업의 결과"를 담는 약속 객체
💡
카페에서 커피를 주문하면 진동벨을 줍니다. 이 진동벨이 바로 Promise입니다. 커피가 완성되면(성공) 벨이 울리고, 재료가 없으면(실패) 직원이 와서 알려줍니다. 받는 순간엔 커피가 없지만 "나중에 알려주겠다"는 약속을 받은 것이죠.

Promise의 3가지 상태

상태의미
Pending (대기)아직 결과가 나오지 않은 초기 상태
Fulfilled (이행/성공)작업이 성공적으로 완료됨
Rejected (거부/실패)작업이 실패함

한 번 결과가 정해지면 다시 바뀌지 않습니다.

Promise 만들고 사용하기

promise-basic.js
const 나의약속 = new Promise((resolve, reject) => {
  const 성공했는가 = true;

  setTimeout(() => {
    if (성공했는가) {
      resolve("작업이 성공했습니다!"); // 성공 시
    } else {
      reject("작업이 실패했습니다.");   // 실패 시
    }
  }, 2000);
});

나의약속
  .then((결과) => console.log("성공:", 결과))
  .catch((에러) => console.log("실패:", 에러));

.then()은 성공(resolve)했을 때, .catch()는 실패(reject)했을 때 실행됩니다.

03

Promise 체이닝

피라미드를 일직선으로 펴는 핵심 원리

핵심 원리는 하나입니다 — .then() 안에서 Promise를 return하면, 다음 .then()이 그 Promise가 끝날 때까지 기다렸다가 실행됩니다.

chaining.js
가져오기_사용자정보("user1")
  .then((사용자) => {
    console.log("사용자 정보:", 사용자);
    return 가져오기_주문목록(사용자.id); // Promise를 return!
  })
  .then((주문목록) => {
    console.log("주문 목록:", 주문목록);
    return 가져오기_주문상세(주문목록[0].id);
  })
  .then((주문상세) => {
    return 가져오기_배송정보(주문상세.id);
  })
  .then((배송정보) => {
    console.log("배송 상태:", 배송정보.status);
  })
  .catch((에러) => {
    // 위 단계 중 어디서든 에러가 나면 여기서 한번에 처리!
    console.log("에러 발생:", 에러);
  });
콜백 지옥Promise 체이닝
오른쪽으로 계속 들여쓰기됨아래로 나란히 이어짐
각 단계마다 에러 처리 필요.catch() 하나로 전체 처리
순서 파악이 어려움위에서 아래로 순서대로 읽힘

자주 하는 실수: return을 빠뜨리는 것

✕ BAD — return 누락
가져오기_사용자정보("u1")
  .then((사용자) => {
    가져오기_주문목록(사용자.id);
    // return 없음!
  })
  .then((주문목록) => {
    console.log(주문목록); // undefined
  });
✓ GOOD — return 명시
가져오기_사용자정보("u1")
  .then((사용자) => {
    return 가져오기_주문목록(사용자.id);
  })
  .then((주문목록) => {
    console.log(주문목록); // 정상 출력
  });
04

async / await

Promise를 동기 코드처럼 읽히게 만드는 문법

async는 함수가 Promise를 반환하는 비동기 함수임을 표시하고, await는 그 Promise가 끝날 때까지 기다렸다가 결과값을 받아옵니다. (async 함수 안에서만 사용 가능)

체이닝 예제를 async/await로

async-await.js
async function 주문정보가져오기() {
  try {
    const 사용자 = await 가져오기_사용자정보("user1");
    const 주문목록 = await 가져오기_주문목록(사용자.id);
    const 주문상세 = await 가져오기_주문상세(주문목록[0].id);
    const 배송정보 = await 가져오기_배송정보(주문상세.id);

    console.log("배송 상태:", 배송정보.status);
  } catch (에러) {
    // try 블록 안 어디서든 에러가 나면 여기서 처리
    console.log("에러 발생:", 에러);
  }
}

주문정보가져오기();
Promise 체이닝async / await
.then()을 계속 이어붙임동기 코드처럼 위→아래로 씀
.catch()로 에러 처리try...catch로 에러 처리
⚠️
await는 반드시 async 함수 안에서만 사용할 수 있습니다. async 함수 밖에서 쓰면 SyntaxError가 발생합니다.
05

fetch API로 실제 서버 통신하기

배운 것을 실전에 적용하는 마지막 단계

fetch는 서버에 HTTP 요청을 보내고 응답을 받는 내장 함수입니다. fetch()는 Promise를 반환하므로 지금까지 배운 방식을 그대로 쓸 수 있습니다.

async/await 방식 (권장)

fetch-get.js
async function 사용자정보가져오기() {
  try {
    const response = await fetch("https://api.example.com/users/1");
    const data = await response.json(); // 이것도 Promise!
    console.log("받아온 데이터:", data);
  } catch (error) {
    console.log("에러 발생:", error);
  }
}
⚠️
가장 많이 하는 실수 — fetch는 서버가 응답만 하면(404, 500이어도) 성공(resolve)으로 처리합니다. 진짜 네트워크 단절일 때만 reject됩니다. 그래서 response.ok를 꼭 확인해야 합니다.
fetch-error-safe.js
async function 데이터가져오기() {
  try {
    const response = await fetch("https://api.example.com/없는페이지");

    if (!response.ok) {
      throw new Error(`서버 에러! 상태 코드: ${response.status}`);
    }

    const data = await response.json();
    console.log(data);
  } catch (error) {
    console.log("에러 발생:", error.message);
  }
}

POST 요청으로 데이터 보내기

fetch-post.js
async function 게시글작성하기() {
  try {
    const response = await fetch("https://api.example.com/posts", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({
        title: "제목입니다",
        body: "내용입니다",
      }),
    });

    if (!response.ok) throw new Error(`요청 실패: ${response.status}`);

    const result = await response.json();
    console.log("작성 완료:", result);
  } catch (error) {
    console.log(error.message);
  }
}
옵션설명
methodHTTP 요청 방식 (GET, POST, PUT, DELETE 등)
headers요청에 대한 부가 정보 (데이터 형식 등)
body실제로 전송할 데이터 (JSON.stringify() 필요)

연습 문제

클릭하면 정답이 펼쳐집니다
Q1.콜백 지옥이 발생하는 근본적인 이유는 무엇일까요?＋
비동기 작업을 순서대로 처리하기 위해 콜백 함수가 계속 중첩되기 때문입니다. 함수 개수나 변수 이름과는 관련이 없습니다.
Q2.Promise의 3가지 상태를 각각 말해보세요.＋
Pending(대기), Fulfilled(이행/성공), Rejected(거부/실패) — 한 번 정해지면 다시 바뀌지 않습니다.
Q3.다음 코드의 문제점을 찾아 고쳐보세요.＋ 
가져오기_데이터("A")
  .then((결과A) => {
    가져오기_데이터("B"); // 문제!
  })
  .then((결과B) => console.log(결과B));
return이 빠져 있습니다. 이 값을 return해야 다음 .then()이 결과를 받을 수 있습니다. 
가져오기_데이터("A")
  .then((결과A) => {
    return 가져오기_데이터("B"); // 해결!
  })
  .then((결과B) => console.log(결과B));
Q4.fetch가 404 에러를 반환해도 .catch()에서 안 잡히는 이유는?＋
fetch는 서버로부터 응답만 받으면 상태 코드와 상관없이 성공(resolve)으로 처리합니다. 해결하려면 response.ok를 확인하고, false라면 직접 throw new Error()로 에러를 발생시켜야 합니다.
Q5.아래 체이닝 코드를 async/await로 바꿔보세요.＋ 
async function 상품리뷰가져오기() {
  try {
    const 상품목록 = await 가져오기_상품목록();
    const 리뷰 = await 가져오기_리뷰(상품목록[0].id);
    console.log(리뷰);
  } catch (에러) {
    console.log(에러);
  }
}
